# Performance Notes

## 2026-07-11 — PERF-001 대상 탐색 비교

### 목적

300개의 이동 적을 기준으로 현재 `selectNearestUniqueTargets`의 bounded selection과 160px 셀 기반 spatial hash를 비교한다.

### 재현

```bash
npm run benchmark:targeting
```

고정 시드로 생성한 적 300개, 탐색 반경 360px, 최대 대상 10명, 2,000회 반복 조건이다. 수치는 실행 장비와 부하에 따라 달라지므로 절대 성능 기준이 아니라 두 방식의 상대 비교로 사용한다.

### 측정 결과

| 방식 | 2,000회 | 현재 방식 대비 |
| --- | ---: | ---: |
| bounded selection | 6.84ms | 기준 |
| spatial hash 조회만 | 4.58ms | 약 33% 빠름 |
| spatial hash 재구축 + 조회 | 38.62ms | 약 5.6배 느림 |

모든 방식은 동일한 최근접 대상 ID와 중복 방지 결과를 반환했다.

### 결정

현재 런타임 대상 탐색을 spatial hash로 교체하지 않는다. 이 게임의 적은 매 프레임 이동하므로 안전한 조회를 위해 인덱스를 계속 재구축해야 하며, 300개 적에서는 재구축 비용이 조회 절감보다 크다. 현재 실제 활성 적 상한도 140개이므로 bounded selection이 더 단순하고 유리하다.

`SpatialHashIndex`는 후속 다중 타워·다중 플레이어처럼 한 프레임에 여러 공간 질의를 공유하는 구조가 생길 때 다시 측정할 수 있도록 실험 구현으로 보존한다.

### 후속 확인

- 실제 Android/iOS 기기에서 100~150개 적의 장시간 FPS와 발열 측정
- 적 상한이 300개를 넘거나 질의 주체가 여러 개로 증가하면 재벤치마크
- 필요 시 `EnemyPool.getActive()`의 배열 할당을 별도 프로파일링

## 2026-07-12 — PERF-002 모바일 장시간 전투

### 목적과 환경

Pixel 5 모바일 Chromium 에뮬레이션에서 100마리 이상 전투를 실제 시간 10분 동안 유지하며 FPS, 활성 객체, 피해 텍스트 풀, JavaScript 힙 변화를 측정한다. 개발 서버와 Phaser 디버그 관찰 지점을 사용하며, 적이 처치될 때마다 5초 간격으로 100마리까지 다시 보충한다. 스테이지 자체 상한에 따라 최대 140마리까지 증가한다.

```bash
npm run test:e2e:endurance
```

일반 `npm run test:e2e`에서는 이 장시간 검사를 건너뛰므로 일상적인 회귀 검사 시간은 늘어나지 않는다.

### 측정 결과

| 항목 | 결과 |
| --- | ---: |
| 실제 실행 시간 | 600초 |
| 5초 간격 표본 | 120개 |
| 평균 FPS | 30.60 |
| 하위 10% FPS | 26.84 |
| 최저 FPS | 26.49 |
| 최대 활성 적 | 140 |
| 최대 활성 투사체 | 1 |
| 최대 활성 피해 텍스트 | 120 |
| 피해 텍스트 풀 최대 크기 | 120 |
| 마지막 스테이지 | 20 |
| JavaScript 힙 순증가 | 0MB |

첫 측정은 평가 오버헤드를 누적하는 상대 대기 방식 때문에 604초 동안 117개 표본만 수집됐다. 이를 절대 시각 기반 120회 샘플링으로 수정한 뒤 두 번째 600초 측정을 통과했다.

### 판단

- 10분 동안 브라우저 종료, 런타임 예외, 풀 크기 무한 증가, 관찰 가능한 힙 순증가는 없었다.
- 피해 텍스트 풀은 설계 상한 120을 지켰다.
- 하위 10% FPS가 30에 미달한 26.84이므로 `PERF-003`에서 프레임별 배열 할당과 활성 객체 순회 비용을 우선 최적화한다.
- 에뮬레이션은 실제 휴대전화의 발열, 배터리 소모, GPU 스로틀링을 재현하지 못하므로 Android/iOS 실기기 검수는 별도로 남긴다.
